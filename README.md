## 📁 The Vault: Repository Structure

Our repository avoids chaotic file dumps by organizing all code and resources into clean, **topic-wise folders**. This makes tracking progress and finding study materials completely frictionless.

```text
├── 📦 01_Arrays/
│   ├── 📄 README.md              
│   └── 🛠️ Two_Sum/
│       └── ... and so on/
|       |
├── 📦 02_Linked_Lists/
├── 📦 03_Stacks_Queues/
|       |
|       |
|       ... and so on/
|       |
├── 📦 09_Trees_Heaps/
└── 📦 12_Dynamic_Programming/

```

### Inside Every Topic Folder:

* **The `README.md`:** Holds the conceptual breakdown of the pattern, curated problem lists (Striver/NeetCode), and hidden edge cases to look out for.
* **The Solutions:** Well-documented source files showcasing multiple optimal approaches to the exact same problem.

## 🤝 How We Collaborate

We follow a clean, collaborative engineering workflow to make sure every line of code added to this repository meets high interview standards.

```text
[Pick Problem] ➔ [Write Solution & Notes] ➔ [Submit PR] ➔ [Peer Code Review] ➔ [Merge to Main]

```

1. **Keep it Sorted:** Add your problem solutions inside the appropriate topic folder. Please don't create unstructured files at the root level.
2. **Code Reviews:** When pushing a new approach, open a Pull Request. We use line-by-line comments to challenge time/space complexities and review variable naming readability.
3. **Use the Discussions:** If your code is failing hidden test cases, don't suffer in silence! Tag it with `status:bug-hunting` and drop it in **The Doubt Box** so someone can unblock you.
